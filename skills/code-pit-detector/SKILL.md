# Code Pit Detector

检查 git commit 或未提交修改中的潜在代码问题（"坑"）。

## 触发方式

- `/code-pit-detector` - 检查当前未提交的修改
- `/code-pit-detector <commit-hash>` - 检查指定 commit

## 工作流程

1. **获取 diff**
   - 无参数：`git diff` + `git diff --cached`
   - 如果无参数且 diff 为空：自动检查最新 commit（`git show HEAD`）
   - 有 commit hash：`git show <commit-hash>`

2. **阅读相关上下文** - 理解修改前后的完整逻辑、相关函数、变量的其他使用位置

3. **分析潜在问题** - 运用判断力识别可能导致 bug 或维护困难的代码模式

4. **输出发现** - 简洁地报告问题并给出修复建议

## 核心原则

- **运用判断力**：不要机械检查，要理解代码意图后判断是否有问题
- **关注真正的风险**：只报告可能导致实际 bug 的问题
- **不要过度防御**：避免建议不必要的防御性编程
- **理解上下文**：在报告问题前，确保理解代码的业务逻辑

## 常见问题类型（仅供参考，不限于此）

以下是一些常见的"坑"类型，但发现问题时不应局限于这些类别：

- **隐式行为不一致** - 参数传入与否导致不同的隐式行为（如排序方向、默认值）
- **调试代码泄露** - 异常的魔法数字、硬编码测试数据、未清理的临时代码
- **边界条件缺失** - 明显需要防护但缺失的场景（时间戳负数、除零等）
- **变量修改未同步** - 重命名/修改后相关引用未更新

详细示例见 [references/pit-examples.md](references/pit-examples.md)。

## 输出风格

**每个问题必须：**

1. **随机选择一种风格**：劈头盖脸/阴阳怪气/老中医把脉/毒舌但专业（详见 [output-styles.md](references/output-styles.md)）
2. **用该风格的一句话点评开头**（不能平铺直叙！）
3. **标注严重性**：🔴 严重 / 🟡 警告 / 🟢 建议
4. **按标准格式输出**：文件路径、问题描述、原代码、diff 修复建议

## 最终步骤：生成 commit

**无论是否发现问题，都在最后执行此步骤（仅限检查未提交修改时）：**

1. 如果未发现问题：给出简短的积极反馈
2. 基于 diff 内容，生成一个合适的 commit message
